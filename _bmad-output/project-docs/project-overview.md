# RoboScope — Project Overview

_Generated by bmad-document-project (scoped deep-dive, 2026-04-14). Focused on inputs for the Phase 4 PRD (OAuth2/SSO + Teams)._

## What this project is

RoboScope is a self-hosted, offline-capable web tool for managing Robot Framework test assets and executions. Teams point it at git repositories, run tests locally or in Docker, compare reports over time, and feed AI-assisted features (spec generation, failure analysis) through LLM providers of their choice. A Chrome extension records browser interactions directly into Robot Framework test files.

## Phase 4 grounding — what already exists

The Phase 4 PRD (OAuth2/SSO + Teams) does not start from zero. Existing primitives:

- **`User` model** — `email`, `username`, `hashed_password` (bcrypt), `role` (single role enum), `is_active`, `last_login_at`. Global role only — no per-project override at the User level.
- **`Role` enum (StrEnum)** — `VIEWER / RUNNER / EDITOR / ADMIN` with integer hierarchy 0–3. Used across global RBAC, API token scope, and ProjectMember role.
- **`ApiToken` model** — SHA256-hashed long-lived token for CI/CD. Has `user_id` + scoped `role` (can be narrower than the user's global role — min-of semantics applied in `_authenticate_api_token`).
- **`ProjectMember` model** — `(user_id, repository_id, role)` unique composite. Per-project role override. Already wired: `/api/v1/repos/{repo_id}/members` CRUD. **This is the key primitive Teams will build on top of.**
- **`AuditLog`** — compliance-grade, auto-captured via middleware. SSO logins and Team membership changes will flow here for free.
- **Auth dependency** `get_current_user` — accepts JWT **or** API token transparently. New SSO sessions can mint the same JWT shape, no new endpoint pattern needed.

## Phase 4 grounding — what's missing

- **No `Team` / `Organization` model** — Teams must be introduced.
- **No identity-provider abstraction** — User has `hashed_password` directly. OAuth2 login requires either a nullable password (user has IdP only) or a separate `UserIdentity` table linking user → provider → external ID.
- **No SAML support** — optional per roadmap (OAuth2 is required; SAML 2.0 is "optional for enterprise IdPs").
- **Hierarchy ambiguity** — currently: global `User.role` → optionally overridden by `ProjectMember.role`. Adding Teams introduces a third layer: `Team.role` inherited by Team members on Team-owned projects. Resolution rule must be decided (max? min? explicit override chain?).
